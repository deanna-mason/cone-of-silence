⏺ The architecture and build plan are done. Two documents now live in your
  (gitignored) planning area:

  - Architecture spec:
  docs/superpowers/specs/2026-07-03-cone-of-silence-architecture.md

  docs/superpowers/plans/2026-07-03-secure-video-chat-phased-plan.md

  The system in one paragraph: Your Next.js app on Vercel grows a real
  /room/[roomId] page. Video flows peer-to-peer in a full mesh of up to 4 —
  there is deliberately no media server, so WebRTC's built-in DTLS-SRTP
  encryption is already end-to-end. One small AWS EC2 box (covered by free
  credits) runs the two backend pieces: a Node.js WebSocket signaling server you
  write yourself (it introduces peers but only relays opaque messages) and
  coturn, a relay that kicks in on restrictive networks and forwards encrypted
  bytes it can't read — that's the "reliable" half.

  The security story is layered, and each layer answers a specific attack: room
  IDs are cryptographically random, and the room secret rides in the URL
  fragment, which browsers never transmit — so your own server can't learn it;
  existing participants (not the server) verify newcomers via HMAC. Then, per
  call, each browser generates an ephemeral keypair and exchanges public keys
  over the WebRTC data channel; from those, every screen derives the same short
  verification code that participants read aloud — a compromised signaling
  server trying to man-in-the-middle the call makes the codes visibly mismatch,
  exactly how Signal's safety numbers work. Your original "opt-in extra
  encryption" idea survives as the stretch phase: a real second cipher layer on
  video frames, wired to the Encryption Toggle you already built.

  Recording is local-only: consent requests travel over the encrypted data
  channels, recording starts only on unanimous accept, everyone sees a
  persistent indicator, anyone can revoke instantly, and the file (.webm) saves
  straight to the recorder's desktop. Nothing uploads anywhere. The spec is also
  honest that we can't stop OS-level screen capture — worth saying in class.

  The seven phases, each independently shippable: (1) camera/mic + room page
  with no networking, (2) signaling server + first two-browser call on
  localhost, (3) AWS deployment with TLS and TURN, (4) 4-person mesh +
  reconnection/reliability, (5) the security layer, (6) consent-gated recording,
  (7) stretch — frame encryption, encrypted chat, adversarial security review.
  Each phase in the plan has agent instructions, files to touch, and concrete
  acceptance criteria, plus standing rules (including the "this Next.js is not
  the one you know — read node_modules/next/dist/docs/ first" warning from
  AGENTS.md).

  One thing only you can do: before Phase 3 you'll need a domain name (~$10/yr)
  and an AWS account — TLS certificates for wss:// and turns:// require a real
  domain. Buying it in week 2 keeps the schedule intact.

  Whenever you're ready, say the word and I'll kick off Phase 1 with an agent.
